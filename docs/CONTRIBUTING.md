# Contributing Guide

This document outlines how to contribute to the Personal Finance App during planning and development phases.

## Development Phases

### Phase 1: Planning & Specification
The project is currently in the planning phase. Contributions focus on:
- Refining product specifications
- Creating architecture and design documents
- Defining data models and APIs
- Identifying edge cases and requirements

### Phase 2: Implementation
Once specifications are finalized, development will begin on:
- Backend implementation
- Frontend implementation
- Database and data layer
- Integration with external services (Plaid, LLM)

## Contributing to Planning

### Before You Start
1. Read all documents in `/docs/product/`
2. Review the [Planning Guide](.github/PLANNING.md)
3. Check the [Domain Glossary](docs/product/10-domain-glossary.md)

### How to Suggest Changes
1. **Product Specification Issues**: Create an issue describing the problem, required capability, or edge case
2. **Design/Architecture Improvements**: Create an issue with detailed rationale
3. **Questions or Clarifications**: Post in Issues with the question

### Document Format
All planning documents should:
- Use Markdown formatting
- Include clear headings and structure
- Provide examples where helpful
- Be concise but complete

## Code Contribution Guidelines (For Implementation Phase)

### Getting Started
1. Fork the repository
2. Create a feature branch: `git checkout -b feature/your-feature`
3. Make your changes
4. Submit a pull request

### Code Standards
- Follow language-specific conventions
- Include tests for new functionality
- Update relevant documentation
- Keep commits atomic and well-described

### Pull Request Process
1. Ensure tests pass
2. Update documentation if needed
3. Get review from maintainers
4. Squash commits if requested
5. Merge when approved

## Reporting Issues

### Bugs
Include:
- Clear description of the bug
- Steps to reproduce
- Expected vs actual behavior
- Environment details

### Feature Requests
Include:
- Clear description of feature
- Use case and motivation
- Proposed API/UI changes if applicable
- Alternative approaches considered

### Questions or Discussions
Use Issues with a `question` label for clarification.

## Code of Conduct

This is a personal project, but contributions should maintain:
- Professional and respectful tone
- Constructive feedback
- Openness to different perspectives
- Focus on shared goals
